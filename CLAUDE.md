# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

`arborium-rt` packages [arborium](https://github.com/bearcove/arborium)'s
plugin runtime as a single emscripten `SIDE_MODULE=2` wasm that loads into
web-tree-sitter's `MAIN_MODULE=2`. One shared runtime instance serves many
per-grammar `SIDE_MODULE` parser wasms dynamically, so the tree-sitter C
runtime and arborium's query runner live once in the browser instead of being
baked into every grammar bundle.

The repo produces one publishable npm package:

- **`@discord/arborium-rt`** ‚ÄĒ the TypeScript runtime wrapper. Ships
  `arborium_emscripten_runtime.wasm` (the Rust SIDE_MODULE ‚ÄĒ built from the
  `arborium-rt-wasm` crate for `wasm32-unknown-emscripten` via `cargo build
  --release`, which emits `arborium_rt_wasm.wasm`; `package wasm host` copies
  it under the historical published filename), the
  `web-tree-sitter.{wasm,mjs}` MAIN_MODULE host (built by
  `./scripts/arborium-rt build wasm host`), every built grammar's wasm + scm
  under `packages/arborium-rt/dist/grammars/<lang>/`, and a single
  generated `dist/grammars.js` that exports `GRAMMARS`. Grammars are
  regenerated by `write-grammars-index.ts` at the end of
  `package wasm grammars`; the host + runtime wasms are staged by `package wasm host`.

A second publishable package targets Node.js:

- **`@discord/arborium-rt-node`** ‚ÄĒ a **statically-linked** native addon
  (built from the `arborium-rt-node` cdylib in `lib/node/` via napi-rs).
  Unlike the browser package, it bundles **every** grammar's
  `parser.c`/scanner compiled directly into the `.node` binary and every
  flattened query baked in as `&'static str` ‚ÄĒ no wasm, no host module, no
  dynamic grammar loading. At first use it registers all grammars into the
  shared `arborium-rt` registry and exposes the same highlight pipeline
  (`highlightToSpans`/`highlightToHtml`/`Session`). Built by
  `./scripts/arborium-rt build node grammars` + `build node`; see "Node native
  addon target" below.

One unpublished workspace package supports development:

- **`@discord/arborium-rt-cli`** ‚ÄĒ the dev CLI. Not published, not built;
  the `./scripts/arborium-rt` wrapper runs it directly from TypeScript
  source via tsx.

## Core commands

All repo tooling funnels through one CLI wrapper: `./scripts/arborium-rt
<subcommand>`. The wrapper runs the CLI's TypeScript source directly via
tsx ‚ÄĒ `pnpm install` on first invocation, then forwards every call. Run
`./scripts/arborium-rt --help` for the full list.

```sh
# First-time setup (populates submodule crates' Cargo.toml, applies patches).
# Re-run any time you bump the submodule or pull new patches.
./scripts/arborium-rt bootstrap

# Build the Rust SIDE_MODULE runtime. The workspace no longer pins a target,
# so the wasm build is explicit: this sets the emscripten target + build-std
# in the environment (needs emcc on PATH). A bare `cargo build` builds only
# the core rlib, natively.
./scripts/arborium-rt build wasm runtime

# Build the MAIN_MODULE host wasm (needs emcc on PATH).
./scripts/arborium-rt build wasm host

# Build one grammar end-to-end (wasm + flattened queries + grammar subdir).
./scripts/arborium-rt build wasm grammar group-acorn json

# Build every grammar in the submodule corpus (pass grammar ids to restrict;
# --group restricts to one arborium group).
./scripts/arborium-rt build wasm grammars [json css]

# Build the statically-linked Node native addon for the HOST triple in two
# steps. `build node grammars` stages every grammar (parser.c + flattened
# queries + manifest) ‚ÄĒ needs the patched tree-sitter CLI (bootstrap), NOT emcc.
# `build node` then links them into one ~80 MB .node ‚ÄĒ needs a C/C++ compiler,
# NOT the tree-sitter CLI. Pass grammar ids to `build node grammars` to restrict
# the set.
./scripts/arborium-rt build node grammars [json markdown html css javascript]  # stage
./scripts/arborium-rt build node                                               # link

# Build + test the TypeScript packages. The arborium-rt-wasm `pretest` hook
# runs `package wasm grammars json` (grammar subdirs) and `package wasm host` (host +
# runtime wasms) into packages/arborium-rt-wasm/dist/ so the Vitest suite
# can actually load them.
pnpm install
pnpm -r build
pnpm -r test

# Run one Vitest file while iterating on the TS wrapper.
pnpm --filter @discord/arborium-rt-wasm test -- arborium.test.mts

# Pack the runtime package tarball (after build wasm grammars + package wasm
# host + package wasm grammars). Releases ship this tarball via GitHub
# Releases, not npm.
pnpm --filter @discord/arborium-rt-wasm pack
```

### Prereqs the tooling expects on PATH

- **emsdk 5.x** (`emcc`, `em++`, `llvm-objdump`). Source `emsdk_env.sh`. CI pins
  **5.0.6** (see `.github/workflows/build.yml`); 4.x is no longer supported. The
  MAIN_MODULE build uses `-O2` to work around a known `-O3` bug where
  `__stack_pointer` is incorrectly eliminated. Grammar SIDE_MODULEs list their
  external-scanner entry points in `EXPORTED_FUNCTIONS` explicitly: the host's
  dynamic linker must resolve the `GOT.func` entries the parser table emits for
  those address-taken functions against the module's own exports, so we name
  them rather than rely on the linker's auto-export heuristic.
- **Nightly Rust** with `rust-src` (for `-Zbuild-std`). A system/nix nightly
  works ‚ÄĒ rustup isn't required.
- **tree-sitter CLI** ‚ÄĒ built locally from the patched submodule by
  `bootstrap` (see "Submodules + patches"). `build wasm grammar` invokes that
  binary, never any system-installed `tree-sitter`, because only the
  patched build understands `TREE_SITTER_SPARSE_ONLY`.
- **Node ‚Č•22, pnpm ‚Č•9**. emsdk 5.x's generated host glue
  (`web-tree-sitter.mjs`) targets a newer Node than 20; under Node 20 its
  WebAssembly GOT symbol resolution returns `undefined` for a required
  symbol and the glue crashes (`Cannot read properties of undefined (reading
  'value')`) when a grammar is loaded. CI runs Node 24.
- **askalono CLI** (`askalono` ‚Č• 0.5). Used by `arborium-rt notices` to
  identify upstream license files via `askalono crawl`. The CLI ships
  with its license-detection cache embedded, so no extra setup is
  required beyond having it on PATH. Install via
  `cargo install --locked askalono-cli` or download a prebuilt binary
  from https://github.com/jpeddicord/askalono/releases.

## High-level architecture

### Runtime library crate (`arborium-rt`, in `src/`)

The Rust runtime is split across two crates in a Cargo workspace. The root
`arborium-rt` crate (an `rlib`, in `src/`) is the target-agnostic core ‚ÄĒ the
registry and highlight pipeline, with no C ABI and no `cdylib` linkage. Both
modules expose `pub` items so the per-target shim crates can drive them:

- **`registry.rs`** ‚ÄĒ single global `Mutex<Registry>` holding every registered
  grammar's `PluginRuntime`. Session IDs are globally unique and routed back
  to the owning grammar. Maintains a `name ‚Üí grammar_id` map used by the
  highlight pipeline's injection resolution. A thread-local `PENDING_LANG`
  cell bridges the `*const TSLanguage` from JS into arborium's `LanguageFn`
  signature (called exactly once during registration, inside the registry's
  mutex ‚ÄĒ no races).
- **`highlight.rs`** ‚ÄĒ the full parse ‚Üí recursive injection ‚Üí dedup ‚Üí
  coalesce ‚Üí theme pipeline. Mirrors `arborium_highlight::HighlighterCore`
  upstream but drops the async `GrammarProvider` because registry lookups
  are just `HashMap` hits. `MAX_INJECTION_DEPTH = 32` is a hard cap against
  pathological grammars. Both the themed-span and HTML outputs render from
  the same resolved span set (`dedup_and_tag` + `coalesce_by_tag`); HTML
  goes through this module's own `tagged_spans_to_html` rather than
  `arborium_highlight::spans_to_html`, because the upstream renderer dedups
  the *raw* span set and mis-resolves the overlapping captures recursive
  injection produces at depth ‚Č• 2 (e.g. markdown's fenced-code literal
  enclosing an injected JS keyword). `tagged_spans_to_html` resolves
  nested spans innermost-wins and skips zero-width spans (whose
  coincident open/close events would otherwise corrupt the render stack).

### WASM shim crate (`arborium-rt-wasm`, in `lib/wasm/`)

The emscripten `SIDE_MODULE=2` target. A thin `cdylib` that depends on the
core `arborium-rt` crate and adds the C ABI:

- **`lib/wasm/src/lib.rs`** ‚ÄĒ the `#[unsafe(no_mangle)] extern "C"` surface
  (formerly `src/abi.rs`). Every `arborium_rt_*` function here must also
  appear in the `EXPORTED_FUNCTIONS` list in `.cargo/config.toml`, or the
  linker will strip it. Pointer rules: input buffers are borrowed; output
  buffers are allocated in shared linear memory and transferred to the
  caller, who frees them via `arborium_rt_free(ptr, len)`. JSON is the wire
  format across this boundary. The shim holds no logic of its own ‚ÄĒ it
  decodes pointers, calls into `arborium_rt::registry` /
  `arborium_rt::highlight`, and serializes the result. This is the layer
  that gets duplicated per target as new (non-wasm) hosts are added;
  the core crate stays target-agnostic.

### Node native addon target (`arborium-rt-node`, in `lib/node/`)

The second per-target shim over the core `arborium-rt` rlib ‚ÄĒ a napi-rs
`cdylib` built for the **host triple**, not emscripten. It has no C ABI of
its own; it calls the same `arborium_rt::registry` / `arborium_rt::highlight`
functions the wasm shim does, but every grammar is **statically linked** in
rather than loaded dynamically.

- **`lib/node/build.rs`** reads the manifest `build node` regenerates from
  the staged grammar dirs (path via `ARBORIUM_RT_NODE_GRAMMARS`,
  default `<repo>/target/node-grammars/manifest.json`). For each grammar it
  `cc`-compiles `parser.c` + the one scanner (`.c` ‚Üí C11, `.cc`/`.cpp` ‚Üí
  C++17 `-fno-exceptions -fno-rtti`) in its **own** `cc::Build` so each
  grammar's `src/` is the only include dir (every grammar ships an identical
  `tree_sitter/parser.h`; per-grammar scoping keeps it unambiguous). It then
  generates `$OUT_DIR/grammar_table.rs`: an `unsafe extern "C"` block of
  `tree_sitter_<cSymbol>()` decls plus a `GRAMMARS` table whose `.scm`
  contents are `include_str!`'d ‚ÄĒ baked into the binary, no runtime fs reads.
- **`lib/node/src/lib.rs`** ‚ÄĒ the napi surface. `grammar_ids()` registers
  every `GrammarDef` once (`OnceLock`), then `highlightToSpans` /
  `highlightToHtmlString` (one-shot) and a `Session` class drive the
  pipeline. The registry is process-global and immutable after init; no
  `unregister` is exposed. `Session` frees its registry session on `Drop`.
- **Staging** lives in TS (the `buildNodeGrammars` step in
  `cli/src/commands/build/node/grammars.ts`), reusing the same `tree-sitter generate`
  + flatten + source-staging helpers as the wasm build ‚ÄĒ extracted into
  `cli/src/lib/stage-grammar.ts` and shared with `cli/src/commands/build/wasm/grammar.ts`.
  The manifest model (`ManifestGrammar`, `manifestEntryFor`,
  `rebuildManifestFromStaged`) lives in `cli/src/lib/node-manifest.ts`, shared
  between the staging (`build node grammars`) and linking (`build node`) halves.
- **Build invocation** (`cli/src/commands/build/node/index.ts`): runs
  `@napi-rs/cli`'s `napi build --platform --release`. `--platform` builds for ‚ÄĒ
  and names the artifact after ‚ÄĒ the runner's own platform, so there is no
  `--target`: each CI matrix entry just runs on a native runner. napi wraps
  `cargo build` for `arborium-rt-node` (`--cargo-cwd lib/node`,
  `--cargo-name arborium_rt_node`), then post-processes: it copies the produced
  cdylib into the npm package as `arborium-rt-node.<platformArchABI>.node`
  (basename from the package's `napi.name`) and regenerates the loader
  `binding.cjs` + types `binding.d.cts` (`--js`/`--dts`) that
  `packages/arborium-rt-node/src/index.ts` imports. The `build node` command
  then *moves* that `.node` into its per-platform sub-package dir,
  `npm/<platformArchABI>/` (see Cross-platform below). No build-std ‚ÄĒ the host
  build links the prebuilt std, and since `.cargo/config.toml` no longer pins
  build-std, there is nothing to clear (setting an *empty*
  `CARGO_UNSTABLE_BUILD_STD` would instead turn build-std on with an empty crate
  set and rebuild `core`, which fails to link). The grammar manifest path is
  handed to `build.rs` via `ARBORIUM_RT_NODE_GRAMMARS`. **This is the only
  supported way to build `lib/node`** ‚ÄĒ it's a workspace `member` but not in
  `default-members` (which is just the core rlib), so a bare `cargo build`
  ignores it; `cargo build --workspace` would (wrongly) try to build both
  shims natively.
- **Cross-platform (standard napi model).** `package.json`'s
  `napi.triples.additional` lists the supported targets (darwin x64/arm64,
  linux x64/arm64 ‚ÄĒ all gnu), and `napi create-npm-dir` generated one
  sub-package manifest per target under `packages/arborium-rt-node/npm/<platform>/`
  (`@discord/arborium-rt-node-<platform>`, each with `os`/`cpu`/`libc` + a
  `main` pointing at its `.node`). The `package-node` CI matrix builds one
  `.node` per platform, each on its own **native** runner (no cross toolchain),
  and stages it into the matching `npm/<platform>/`; the generated `binding.cjs`
  loader `require`s `@discord/arborium-rt-node-<platform>` at runtime, gated by
  `process.platform`/`arch`. The main package depends on the four sub-packages
  via **`workspace:*` optionalDependencies**, and the `npm/*` dirs are pnpm
  **workspace members** (see `pnpm-workspace.yaml`). That is the crux: pnpm
  resolves them locally (`link:npm/<platform>`), so `pnpm install
  --frozen-lockfile` works with **nothing published** to the registry ‚ÄĒ no
  bootstrap publish needed. `os`/`cpu`/`libc` on each sub-package gate which one
  actually installs per platform (exactly how `fsevents` works). The committed
  `npm/<platform>/package.json` files are tracked; the built `.node` inside them
  is gitignored (`*.node`). At release, `pnpm publish` rewrites `workspace:*` to
  the real version and publishes the main package + the four sub-packages.

### Node consumer package (`packages/arborium-rt-node/`)

Thin TS wrapper (`src/index.ts` + `src/types.ts`) over the napi addon,
published as `@discord/arborium-rt-node`. Imports the native functions +
`Session` from the napi-rs-generated loader `../binding.cjs` (a CJS module the
ESM wrapper imports by name; built by `build node`), translates the
`HtmlFormat` object union into the numeric format code the addon expects, and
re-exports types kept
structurally identical to `packages/arborium-rt/src` (`ThemedSpan`,
`HighlightSpansResult`, `HighlightHtmlResult`, `HtmlFormat`, `HighlightOptions`)
so the two packages are interchangeable for highlighting. Its `pretest`
builds the addon with a small positional grammar set for a fast Vitest loop.

### TypeScript consumer package (`packages/arborium-rt/`)

Typed `Runtime` / `Grammar` / `Session` API over the ABI, published as
`@discord/arborium-rt`. `loadArboriumRuntime()` takes no args ‚ÄĒ the
host `.mjs`/wasm and the runtime wasm are staged into
`packages/arborium-rt/dist/` (under `host/` and `runtime/`) by
`arborium-rt package wasm host`, and the TS resolver uses
`new URL('./host/‚Ä¶', import.meta.url)` so bundlers trace the specifiers.
`loadArboriumRuntime` also pre-reads the host wasm bytes via `resolveWasm`
and passes them to the factory as `Module.wasmBinary` ‚ÄĒ required because
the host is built with `-sENVIRONMENT=web,worker` (no Node branch, so no
auto `fs`-based loader; see the `build wasm host` notes below).

Bundled grammars live under `dist/grammars/<lang>/` (each subdir is just
the `tree-sitter-<lang>.wasm` + flattened `.scm` files; no per-grammar
`index.js`). Metadata for the whole set lives in a single generated
`dist/grammars.js` that exports `GRAMMARS` ‚ÄĒ every language keyed by id,
URLs pointing at the sibling subdirs. Re-exported from the package root
so consumers only need `import { GRAMMARS } from '@discord/arborium-rt'`.
`src/grammars.ts` is gitignored and regenerated by
`write-grammars-index.ts` whenever `arborium-rt package wasm grammars`
runs; `package wasm host` + at least one `package wasm grammars` run is the
`pretest` for this package.

Per-language modules under `./grammars/*` no longer exist and the subpath
export was removed ‚ÄĒ the single-object design made the per-lang duplication
redundant. `LoadGrammarOptions.highlights / injections / locals` accept
either a string or a URL; the runtime resolves URLs via the same
`fetch`/`readLocal` path as `wasm`. The raw ABI shape lives in `src/abi.ts`;
the user-facing wrappers in `src/runtime.ts`.

The runtime emits *tagged* spans via `arborium_theme::tag_for_capture` ‚ÄĒ
short slot strings like `"k"`, `"f"`, `"s"` that consumers map to colors
themselves. arborium-rt does **not** bundle any theme CSS; styling is
the consumer's responsibility.

### Dev CLI (`cli/`)

Private, unpublished workspace package. Run directly from source via tsx
through `./scripts/arborium-rt`. Entry point is `src/index.ts` (sade command
table); each subcommand lives under `src/commands/`, grouped by verb then
platform. The wasm-platform commands live under `build/wasm/` and
`package/wasm/`: `build/wasm/{runtime,host,grammar,grammars}.ts` (‚Üí `build wasm
runtime`, `build wasm host`, `build wasm grammar <group> <lang>`, `build wasm
grammars`) and `package/wasm/{host,grammars}.ts` (‚Üí `package wasm host`,
`package wasm grammars`). The Node addon lives under `build/node/`:
`build/node/grammars.ts` (‚Üí `build node grammars`, staging) and
`build/node/index.ts` (‚Üí `build node`, linking). Truly cross-cutting commands stay at the top
level: `bootstrap`, `apply-patches`, `list-groups`, `notices`. (Note: sade
routes a 3-token command like `build wasm runtime` only because no shorter
`build` command is registered to short-circuit the lookup ‚ÄĒ if you ever add a
bare `build` command back, the `build wasm *` group breaks unless a `build wasm`
parent command is *also* registered. `index.ts` keeps a `build wasm` parent
regardless, purely so `build wasm` with no subcommand prints the subcommand
help.) Shared helpers in `src/lib/util.ts`
(repo-root discovery via walking up for a `Cargo.toml` whose package line is
`name = "arborium-rt"` ‚ÄĒ matched with the closing quote so it doesn't also
hit `lib/wasm`'s `arborium-rt-wasm` manifest; overridable via
`ARBORIUM_RT_ROOT`). The `build wasm host` module
(`src/commands/build/wasm/host.ts`)
owns the two "extras" arrays (`EXTRA_TS_EXPORTS`, `EXTRA_LIBC_EXPORTS`) that
expand upstream's export lists with plain `ts_*` symbols and libc/pthread
calls Rust's `std` pulls in ‚ÄĒ these are the source of truth until upstream
takes them. The host emcc call uses `-sENVIRONMENT=web,worker`; omitting that
re-introduces the Node branch (`await import("module")`, bare
`require("fs"|‚Ä¶)`) that rspack static-rejects. `src/lib/write-grammars-index.ts`
regenerates the wasm package's `src/grammars.ts` after `package wasm grammars`;

### Submodules + patches

Two pinned submodules live under `third_party/`:

- **`third_party/arborium`** ‚ÄĒ pinned to a specific commit (commit, not
  tag ‚ÄĒ `arborium-plugin-runtime` doesn't exist on any released tag yet).
  The root `Cargo.toml` path-deps four crates out of it:
    - `arborium-plugin-runtime` ‚ÄĒ unpatched; unpublished upstream.
    - `arborium-tree-sitter` ‚ÄĒ **patched** to skip static tree-sitter C
      linking on emscripten (the MAIN_MODULE resolves those symbols).
    - `arborium-wire` ‚ÄĒ unpatched; unpublished upstream.
    - `arborium-highlight`, `arborium-theme` ‚ÄĒ consumed with
      `default-features = false` by the emscripten runtime. Only
      `tag_for_capture` is used from `arborium-theme`; the `toml`
      feature (theme parsing) is never enabled.
- **`third_party/tree-sitter`** ‚ÄĒ pinned to tag `v0.26.8`. Vendored so
  we can build a **patched** `tree-sitter` CLI that emits sparse-only
  parser tables. The patch (`patches/tree-sitter/0001-sparse-only-render.patch`)
  adds a `TREE_SITTER_SPARSE_ONLY` env-var check in
  `crates/generate/src/render.rs::populate_used_symbols_and_state_count`
  that forces `large_state_count = 0`, plus two conditionals around
  `add_parse_table` and the `TSLanguage` struct emit so the dense
  `ts_parse_table[LARGE_STATE_COUNT][SYMBOL_COUNT]` array isn't written.
  `build wasm grammar` runs the bootstrap-built binary at
  `third_party/tree-sitter/target/<host>/release/tree-sitter` with the
  env var set, **never** the system `tree-sitter`. Cuts the SIDE_MODULE
  wasm by 50‚Äď78% on the largest grammars (verilog 17.3 ‚Üí 4.4 MB,
  cobol 11.8 ‚Üí 2.5 MB, lean 17.3 ‚Üí 8.7 MB, sql 9.4 ‚Üí 3.7 MB) at the
  cost of ~20% parse-time slowdown on big files ‚ÄĒ fine for highlighting,
  which is one-shot per file. The arborium-tree-sitter runtime already
  handles `large_state_count == 0` (every lookup goes through the
  sparse path; cf. `crates/arborium-tree-sitter/src/language.h:78`).

Patches live as mbox files under `patches/<submodule>/` (`git
format-patch` output). All current patches are trivial target guards or
opt-in codegen flags ‚ÄĒ no logic changes to existing behavior.

`./scripts/arborium-rt bootstrap` is **idempotent**: it resets each
submodule to its pinned SHA, `git apply`s every patch under
`patches/<submodule>/` to the working tree (no commits ‚ÄĒ `git apply`
ignores the mbox preamble, so no committer identity is required),
renders every arborium `Cargo.stpl.toml` ‚Üí `Cargo.toml`, writes a stub
`crates/arborium-theme/src/builtin_generated.rs` (gitignored upstream,
normally produced by `cargo xtask gen`; arborium-theme's `theme.rs`
`include!`s it unconditionally so the file must exist ‚ÄĒ arborium-rt
ships an empty `pub fn all() -> Vec::new()` since no themes are
bundled), and builds the patched tree-sitter CLI with an explicit
`CARGO_BUILD_TARGET=<host>` (a host-native CLI binary, independent of
whichever target a later `build wasm runtime`/`build node` selects). Re-run
bootstrap after bumping either submodule or tweaking a patch.

### Target layout

- `target/wasm32-unknown-emscripten/release/arborium_rt_wasm.wasm`
  ‚ÄĒ Rust SIDE_MODULE output (~1.1 MB uncompressed), emitted by the
  `arborium-rt-wasm` cdylib. `package wasm host` copies it into the runtime
  package's `dist/runtime/arborium_emscripten_runtime.wasm` (the published filename).
- `target/host-wasm/web-tree-sitter.{wasm,mjs}` ‚ÄĒ MAIN_MODULE host.
- `target/grammars/<lang>/` ‚ÄĒ `tree-sitter-<lang>.wasm` +
  flattened `.scm` query files (written by `build wasm grammar`).
- `packages/arborium-rt/dist/grammars/<lang>/` ‚ÄĒ per-grammar assets
  (the wasm + flattened `.scm` files). Written by `package wasm grammars`;
  referenced by the generated `dist/grammars.js` via `new URL(...)`.
- `packages/arborium-rt/dist/grammars.js` ‚ÄĒ single `GRAMMARS` aggregator.
  Generated by tsc from `src/grammars.ts`, which is itself generated by
  `write-grammars-index.ts` after `package wasm grammars`. Exposed to
  consumers as `import { GRAMMARS } from '@discord/arborium-rt'`.
- `target/node-grammars/<lang>/` ‚ÄĒ per-grammar staging for the Node addon:
  `src/parser.c` + `src/tree_sitter/*.h` + optional scanner + flattened
  `.scm`, written by `build node grammars`.
  `target/node-grammars/manifest.json` is the index `lib/node/build.rs` consumes.
- `target/<triple>/release/libarborium_rt_node.{dylib,so}` ‚ÄĒ the native addon
  for the host `<triple>`. `napi build` copies + renames it to
  `arborium-rt-node.<platformArchABI>.node` in the package dir, then `build node`
  moves it into its sub-package at
  `packages/arborium-rt-node/npm/<platformArchABI>/arborium-rt-node.<platformArchABI>.node`.
  The generated `binding.cjs`/`binding.d.cts` loader + types land in the main
  package dir. The `.node` files and `binding.*` are gitignored; the
  `npm/<platform>/package.json` manifests are committed.

### Verifying a runtime build

```sh
/path/to/emsdk/upstream/bin/llvm-objdump --syms \
```

Expects all ten entry points: `arborium_rt_register_grammar`,

### Grammar build gotchas

- **Query inheritance** (e.g. HLSL ‚Üí C++ ‚Üí C) is resolved transitively by
  reading each grammar's `arborium.yaml` `queries.highlights.prepend`
  list and concatenating; matches arborium's own Rust-plugin template.
  The flattener is in `cli/src/lib/flatten.ts`.
- **C++ external scanners** (`scanner.cc` / `scanner.cpp`) are compiled
  with `em++ -std=c++17 -fno-exceptions -fno-rtti` and linked alongside
  `parser.c`. No arborium grammar uses this today, but the path is
  scaffolded.
- **`grammar.js` that `require()`s other grammars** (TSX pulls in
  `tree-sitter-javascript`, SCSS pulls in `tree-sitter-css`, etc.)
  needs those npm packages staged into `node_modules/` before
  `tree-sitter generate` runs. `build wasm grammar` does this by symlinking
  vendored dep grammars' `def/grammar/` dirs into the build dir and
  exposing `NODE_PATH` ‚ÄĒ but only for grammars already in the
  arborium corpus. Third-party upstream deps are not yet wired up.
- **Sparse-only parser tables.** `build wasm grammar` always sets
  `TREE_SITTER_SPARSE_ONLY=1` when invoking the patched tree-sitter CLI.
  The arborium-tree-sitter runtime treats that as the supported case ‚ÄĒ
  every state lookup goes through `ts_small_parse_table` (cf.
  `crates/arborium-tree-sitter/src/language.h:78`). To compare against
  the dense layout, point `build wasm grammar` at an unpatched binary and
  unset the env var; the wasm will be 2‚Äď4√ó larger but the parse tree
  will be byte-identical.

## Conventions worth knowing

- **`cargo build` builds the core rlib natively; each target is explicit.**
  `.cargo/config.toml` no longer pins a default target ‚ÄĒ it only carries the
  `[target.wasm32-unknown-emscripten]` link args, which apply when that target
  is selected. `default-members = ["."]` in the root Cargo.toml means a bare
  `cargo build` compiles only the target-agnostic `arborium-rt` rlib for the
  host. The two deployable shims are built through the CLI, each setting its
  own target in the environment:
    - `arborium-rt build wasm runtime` ‚Üí `arborium-rt-wasm` for
      `wasm32-unknown-emscripten` with `CARGO_BUILD_TARGET` +
      `CARGO_UNSTABLE_BUILD_STD=std,panic_abort` (the `SIDE_MODULE`/
      `EXPORTED_FUNCTIONS` link-args come from `.cargo/config.toml`). Needs emcc.
    - `arborium-rt build node` ‚Üí `arborium-rt-node` for the host triple with
      `CARGO_BUILD_TARGET=<host>` (no build-std; the host links prebuilt std).
  Never add `lib/wasm`/`lib/node` back to `default-members`: a default build
  would then try to compile a shim for the wrong target.
- **Never edit `third_party/arborium/crates/*/Cargo.toml`** ‚ÄĒ they're
  generated from `.stpl.toml` templates by bootstrap. Edit the template or
  patch the template-rendering step in `bootstrap.ts` instead.
- **Adding a new `arborium_rt_*` export requires three coordinated edits**:
  the function in `lib/wasm/src/lib.rs`, its name in `.cargo/config.toml`'s
  `EXPORTED_FUNCTIONS` list, and the TS binding in
  `packages/arborium-rt/src/abi.ts` (plus a method on the wrapper class if
  user-facing). Forgetting any of these produces confusing runtime errors,
  not compile errors.
- **The Node addon is the parallel surface.** A new highlight capability
  usually wants both a `#[napi]` function/method in `lib/node/src/lib.rs`
  (driving the same `arborium_rt::*` calls) and a matching wrapper +
  type in `packages/arborium-rt-node/src/{index,types}.ts`. `napi build`
  generates the JS loader + `.d.ts` from the macro (no `EXPORTED_FUNCTIONS`
  list) into `binding.cjs`/`binding.d.cts`, which the wrapper imports; rerun
  `build node` to regenerate them after changing the `#[napi]` surface. napi
  camelCases Rust `snake_case` struct fields and arg names, so keep the TS
  types camelCase to match.
- **Adding a wasm `extern`** (e.g., pulling in a new libc or tree-sitter
  symbol) requires adding it to `EXTRA_TS_EXPORTS` or `EXTRA_LIBC_EXPORTS`
  in `cli/src/commands/build/wasm/host.ts` and rebuilding the host.
  Discover missing imports with `wasm-dis arborium_rt_wasm.wasm
  | grep '(import "env"'`.
- **Bumping the submodule**: checkout new SHA in `third_party/arborium`,
  re-run `./scripts/arborium-rt bootstrap`. If patches no longer apply,
  `git apply` rejects the patch and leaves the working tree
  partially-modified ‚ÄĒ reset with `git -C third_party/arborium reset
  --hard && git -C third_party/arborium clean -fd`, fix the offending
  `patches/*.patch` file, and re-run. Then rebuild the affected targets
  (`./scripts/arborium-rt build wasm runtime` and/or `build node`) to confirm they
  still link, and commit both the submodule bump and any patch updates together.
